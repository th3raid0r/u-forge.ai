# Feature: Schema Editor & Migrator

## Context

Schemas in u-forge.ai currently can't evolve. The in-memory `SchemaDefinition`
type and the `.schema.json` files in `defaults/schemas/` are both shaped after
Memorymesh's MCP conventions (e.g. `add_` prefix stripping, `metadata: ["k:v"]`
strings) — there is no native u-forge schema format. Nodes carry an optional
`schema_name: String` but no version, so once a schema changes the old records
are silently orphaned against the new definition; no validation, no migration,
no way to re-apply. The `version: String` field on `SchemaDefinition` exists
but is never read.

The goal is a first-class schema lifecycle: a native u-forge schema format,
real per-schema versioning, an authoring UI (spreadsheet-style per node type),
and a migration planner that produces a shareable artifact and executes against
existing data.

## Scope split

- **Phase 0 — Infrastructure:** native schema format, version tracking on
  schemas and nodes, comparison/diff primitive, migration-plan artifact format,
  migration engine with transactional apply + chunk invalidation, removal of
  hardcoded default schema, CLI/test harness.
- **Phase 1 — UI:** edit-icon entry point, per-type schema-edit tab with
  `Stage changes`, session staging layer, yellow staging banner, multi-step
  migration wizard modal (shared by the banner and the import-conflict flow),
  enhanced import dialog with optional migration-plan field.

## Design decisions

1. **Drop Memorymesh as the on-disk format.** Native on-disk format is ours;
   output files are `.uforge-schema.json`. `convert_memorymesh.rs` today only
   converts *data* (JSONL, dropping unknown fields) — it does **not** convert
   schemas. Extend it to also read the Memorymesh-shaped `.schema.json` files
   in `defaults/schemas/` and emit native `.uforge-schema.json` files. Same
   binary, two modes (subcommands or flags).

2. **Versioning model.** A schema is identified by a stable `schema_id: Uuid`
   that persists across versions, plus a monotonic `version: u32` (drop the
   semver string). Each saved schema row is a `(schema_id, version)` pair; we
   keep every version, never overwrite. A `parent_version: Option<u32>` field
   records lineage. `name` becomes display-only.

3. **Per-node version tracking.** Add `schema_version INTEGER` to `nodes`
   alongside the existing `schema_name`. Migrations mark nodes with the new
   version after successfully transforming their properties. Nodes without a
   schema (`NULL`) are untouched.

4. **Migration plan as a first-class artifact.** Saved as
   `.uforge-migration.json`. Self-contained: source `(schema_id, version)`,
   target `(schema_id, version)`, and a per-object-type ordered list of
   operations. Shareable across users/sessions.

5. **Property operations.** The op depends on whether the destination property
   already carries v1 data:
   - `Delete { property }` — drop the v1 key. No destination.
   - `Move { from, to }` — destination is **new** in v2 (didn't exist in v1).
     Plain rename into an empty slot.
   - `Append { from, to, separator }` — destination **existed in v1**; v1
     value is string-concatenated onto it (e.g. fold an old field into
     `description`). Default separator `"\n\n"`.
   - `Replace { from, to }` — destination existed in v1; v1 value overwrites
     it.
   - Implicit op: `AddWithDefault { property, default }` for new v2 properties
     that don't correspond to any v1 field. Auto-generated by the diff engine,
     not selectable from the action dropdown.

   **UI column order:** `source (v1) | destination (v2) | action`. Column 2 is
   picked first; it gates column 3's options. Destination is new-in-v2 →
   `[Move, Delete]`; destination existed in v1 → `[Append, Replace, Delete]`;
   destination is `(none)` → `[Delete]`.

6. **Migration engine stays in `u-forge-core`.** Hangs off `KnowledgeGraph`
   as `migrate_schema(plan) -> MigrationReport`. Transactional: wrap the whole
   apply in `BEGIN IMMEDIATE` / `COMMIT`, abort on any node that fails
   validation against the target schema (strict by default; lenient as opt-in
   flag in `MigrateOptions`).

7. **Chunk + embedding invalidation.** After transforming a node's properties,
   delete its chunks (FTS5 auto-syncs via triggers) and re-flatten via the
   existing `flatten_for_embedding` path. Re-embedding is left to the normal
   async `InferenceQueue` pass — migration doesn't block on it. The staging
   model (§1.3) ensures users can edit multiple types in one session and pay
   this cost exactly once.

### 0.4 Migration plan format

**New module:** `crates/u-forge-core/src/schema/migration.rs`

```rust
pub struct MigrationPlan {
    pub source: SchemaRef,  // { schema_id, version }
    pub target: SchemaRef,
    pub object_type_ops: HashMap<String, Vec<PropertyOp>>,
    pub created_at: DateTime<Utc>,
    pub description: Option<String>,
}

pub enum PropertyOp {
    Delete         { property: String },
    Move           { from: String, to: String },
    Append         { from: String, to: String, separator: String },
    Replace        { from: String, to: String },
    AddWithDefault { property: String, default: serde_json::Value },
}
```

Helpers:

- `MigrationPlan::from_diff(diff, source, target)` — auto-populates `Delete`
  and `AddWithDefault` ops; leaves removed→added pairings as unresolved for
  the user to assign via UI. Used by the wizard and import-conflict flow.
- `MigrationPlan::write_to_file(path)` / `read_from_file(path)` — JSON
  serialization. The shareable artifact.
- `MigrationPlan::validate_against(source_schema, target_schema)` — catches
  dangling property references before apply.

### 0.5 Migration engine

**New method on `KnowledgeGraph` in `crates/u-forge-core/src/lib.rs`:**

```rust
pub async fn migrate_schema(
    &self,
    plan: &MigrationPlan,
    target_schema: &SchemaDefinition,
    opts: MigrateOptions,
) -> Result<MigrationReport>;
```

Steps:

1. Validate plan against source + target schemas.
2. `BEGIN IMMEDIATE` transaction.
3. Upsert target schema at new `(schema_id, version)`.
4. Select all nodes where `schema_name = source.name AND schema_version =
   source.version`.
5. For each node: apply `PropertyOp`s to the JSON blob in order, validate
   against target schema, update row with new `schema_version`.
6. `delete_chunks_for_node(id)` for each touched node (FTS5/vec0 cascade via
   existing triggers — no extra work needed).
7. `COMMIT`. Return `MigrationReport { migrated, skipped, warnings }`.
8. After commit, re-flatten touched nodes + enqueue for embedding via the
   existing `InferenceQueue` path.

Other files to wire:
- `crates/u-forge-core/src/types.rs` `ObjectMetadata` — add
  `schema_version: Option<u32>`.
- `crates/u-forge-core/src/graph/nodes.rs` `upsert_node` — already supports
  the update path, no changes needed.
- `crates/u-forge-core/src/ingest/pipeline.rs` `flatten_for_embedding` —
  reused as-is.

### 0.6 Remove hardcoded default schema

`SchemaDefinition::create_default()` and all `ObjectTypeSchema::default_*` /
`EdgeTypeSchema::default_*` factories in `definition.rs` (lines 87–118+) are
Memorymesh-era training wheels. `pipeline.rs` already explicitly deletes the
"default" schema after loading real files (~line 114), confirming it's treated
as pollution.

- Delete every `default_*` factory and `create_default()`.
- `SchemaIngestion` at startup looks for `./defaults/schemas/` relative to the
  process working directory (keep the existing `UFORGE_SCHEMA_DIR` env
  override). If the directory is empty or absent, the app launches with no
  schemas — the UI handles it.
- Update any test that relied on `create_default()` to build its fixture schema
  explicitly.

### 0.7 Extend `convert_memorymesh` for schemas

**File:** `crates/u-forge-core/examples/convert_memorymesh.rs`

Today this tool converts data JSONL only, dropping unknown fields. Add a schema
conversion mode that reads the Memorymesh-shaped `.schema.json` files in
`defaults/schemas/` (with `add_` prefixed names, `metadata: ["k:v"]` strings,
etc.) and emits native `.uforge-schema.json` files. Same binary, separate
subcommand or `--schemas` flag. Verify the output round-trips through
`SchemaIngestion::load_schemas_from_directory`.

### 0.8 Test harness

`crates/u-forge-core/tests/schema_migration.rs`: end-to-end tests loading a
fixture schema v1, inserting nodes, applying a synthetic plan, asserting the
post-migration property shape. Must exercise every `PropertyOp` variant,
including the string-concat behavior of `Append` and the rollback path when
strict validation fails. Run under `cargo test --workspace -- --test-threads=1`.

## Phase 1 — UI

Editing and migration are split so a user can revise multiple node types in one
session and pay the re-embedding cost once.

- **Editing** — a tab in the main editor view, one per object type.
- **Staging** — session-wide layer accumulating edits across types.
- **Migration planning** — a wizard modal launched from a global banner,
  operating on all staged changes atomically.

### 1.1 Entry point — edit icon in the node panel

**File:** `crates/u-forge-ui-gpui/src/node_panel.rs`

The node panel groups nodes by object type and already has a `+` button per
group. Add a pencil icon (`✎`) to the left of each `+`:

- `+` — create a new node (existing behavior).
- `✎` — open a schema-edit tab for this object type in the main editor view.

Clicking `✎` for a type that already has an open tab focuses that tab instead
of opening a second one.

### 1.2 Schema-edit tab (per object type)

A new tab kind in the main editor view, peer to node editor tabs — not a
sub-tab of any node's editor.

Contents:

- Spreadsheet-like property table: rows are properties, columns
  `name | type | required | description | validation`. Reuse `TextFieldView`
  for inline editing and the enum-dropdown pattern
  (`node_editor/render.rs:318-525`) for `PropertyType`. Add Row / Delete Row
  at the bottom.
- Footer: single primary button `Stage changes`. On press, diff the tab's
  working copy against the currently-saved `ObjectTypeSchema` and push the
  per-type diff into the session staging layer. Tab stays open.

Edits before `Stage changes` are local to the tab — they don't touch SQLite
and are discarded if the tab is closed without staging.

### 1.3 Session staging layer

**New module:** `crates/u-forge-ui-gpui/src/schema_staging.rs`

```rust
pub struct StagedSchema {
    pub schema_id: Uuid,
    pub base_version: u32,
    pub staged_object_types: HashMap<String, ObjectTypeSchema>,
}
```

Stored on `AppView`. Populated by `Stage changes` (§1.2). Cleared on
successful migration apply or explicit Discard.

### 1.4 Yellow staging banner

When `staged_schema` is non-empty, render a yellow bar overlay immediately
below the menubar, spanning the editor and L/R panels (same z-layer as
modals, below the menubar):

- Left: `N type(s) have staged schema changes — v{base} → v{base+1}`
- Right: `[Plan migration]` `[Discard]`

### 1.5 Migration wizard modal

Opened by `Plan migration` on the banner, and also by the import-conflict
path (§1.6) — one implementation, two entry points.

**Step 1 — Review.**

Left rail: list of staged object types with `+N / -N / ~N` badges. Right
pane: 3-column migration table scoped to the selected type.

| source (v1 property) | destination (v2 property) | action |
|---|---|---|
| read-only name + type | dropdown: v2 properties + `(none)` | contextual |

Action options by destination:
- `(none)` → `[Delete]`
- new-in-v2 → `[Move, Delete]`
- existed in v1 → `[Append, Replace, Delete]`

Below the main rows, an `AddWithDefault` section for v2-only properties
with editable default values.

**Step 2 — Confirm.**

Summary: total nodes affected across all staged types, chunk invalidation
count, warning that re-embedding queues after apply.

**Footer actions** (both steps):
- `Export plan…` — writes `.uforge-migration.json` spanning all staged types.
- `Apply migration` — single `migrate_schema` call; on success, clear staging
  and dismiss.
- `Back to editing` — dismiss modal, preserve staging.

**New module:** `crates/u-forge-ui-gpui/src/migration_wizard/`

### 1.6 Enhanced import dialog

**File:** `crates/u-forge-ui-gpui/src/app_view/mod.rs`
**Function:** `do_import_schema_picker`

Replace the single-path `PathPickerModal` with a two-field modal:

1. Schema directory (required)
2. Migration plan file (optional)

On confirm:

- Load the incoming schema.
- No existing schema with this `schema_id` → save, done.
- Existing schema matches exactly (`diff_schemas` is identical) → toast
  "Schema already imported" and bail.
- Different + plan provided → validate plan against both schemas, apply
  directly. No wizard needed — the artifact is authoritative.
- Different + no plan → seed `StagedSchema` from the incoming schema's object
  types and open the migration wizard (§1.5) pre-filled with the full diff.

## Verification

**Phase 0 (headless):**

1. `cargo test --workspace -- --test-threads=1` — `schema_migration.rs`
   covers: identical-schema detect; `Move`/`Append`/`Replace`/`Delete`;
   `AddWithDefault`; strict rollback on validation failure; chunks
   re-flattened after apply.
2. CLI sanity: fixture schema v1, 10 npc nodes, hand-authored
   `.uforge-migration.json`, run via `examples/migrate_schema.rs`. Confirm
   `nodes.schema_version` updates to 2 for all rows and `chunks` contain
   fresh content.
3. Converter: run extended `convert_memorymesh` against `defaults/schemas/`
   (Memorymesh-shaped), confirm `.uforge-schema.json` output round-trips
   through `SchemaIngestion::load_schemas_from_directory`. Confirm existing
   data-conversion mode still works.

**Phase 1 (UI):**

1. Launch with no schemas — app starts clean (no hardcoded default). Place
   files in `./defaults/schemas/` → they load on startup.
2. Import a fresh schema → saved as v1, no migration prompt.
3. Click `✎` next to `npc` → schema-edit tab opens. Rename a property, press
   `Stage changes`. Yellow banner: "1 type has staged schema changes". Repeat
   for `location`. Banner: "2 types…".
4. Click `Plan migration` → wizard opens with both types and badges. 3-col
   table shows the rename as a `Move` row. `Apply migration`. Banner clears.
   Verify exactly one re-embedding pass (check `get_unembedded_chunks` count
   before/after).
5. Stage again, `Export plan…` → `.uforge-migration.json` contains ops for
   both types.
6. Import an identical schema → toast, no state change.
7. Import a different multi-type revision with no plan → staging seeded from
   incoming schema, wizard opens pre-filled. Same apply path as (4).
8. Import same revision with its `.uforge-migration.json` → applies directly,
   no wizard, one re-embedding pass.

## Open questions

- **Schema-level vs object-type-level versioning.** Plan versions the whole
  schema. Per-type versioning would allow finer-grained partial migrations but
  adds complexity. Recommendation: whole-schema for now.
- **Legacy DB compat.** The open-time migration in §0.2 handles existing dev
  databases. If it's safe to assume no one has data worth preserving, we can
  skip it and require a fresh DB instead.
- **Strict vs lenient default.** Proposal: strict (abort on first failure) by
  default, lenient as a `MigrateOptions` flag.
